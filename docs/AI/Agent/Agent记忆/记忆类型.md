# 记忆类型

## 1. 记忆的工程语境

### 1.1 记忆解决的任务问题

Agent 记忆用于保存当前任务进展和跨任务经验。没有记忆，Agent 只能依赖当前上下文窗口；任务变长后，早期约束、关键决策和失败原因会被挤出上下文。记忆模块让 Runtime 能把重要信息保存下来，并在合适时机重新注入模型输入。

记忆不等于完整聊天历史。聊天历史只是原始材料，工程上需要抽取、压缩、索引和更新。哪些内容值得保存，保存多久，何时读取，谁可以访问，都属于记忆设计。

### 1.2 分类口径

不同资料对记忆分类并不完全一致。工程上可以按生命周期和内容类型组合理解。

| 类型 | 保存内容 | 生命周期 | 常见载体 |
| --- | --- | --- | --- |
| 短期记忆 | 当前任务状态、最近消息、工具观察 | 单次任务 | 上下文窗口、状态对象 |
| 长期记忆 | 用户偏好、业务事实、历史经验 | 跨任务 | 向量库、数据库、文件 |
| 情景记忆 | 某次任务的轨迹和结果 | 中长期 | trace、摘要、案例库 |
| 语义记忆 | 稳定知识和概念 | 长期 | 知识库、文档、图数据库 |
| 过程记忆 | 完成任务的方法和步骤 | 长期 | Skill、流程模板、失败经验 |
| 实体记忆 | 人、项目、系统、订单等实体事实 | 长期 | 结构化表、图谱 |

这几类可以同时存在。例如一次代码修复任务的短期状态记录当前文件和测试结果；任务完成后，失败原因可以进入情景记忆；修复步骤可以沉淀成过程记忆。

## 2. 短期记忆

### 2.1 状态对象

短期记忆应从消息历史中拆出来。状态对象可以包含目标、阶段、已用工具、已读文件、证据、错误、预算和完成条件。模型每轮只需要看到压缩后的状态摘要。

```json
{
  "goal": "修复登录页报错",
  "phase": "test_fix",
  "read_files": ["src/login.ts"],
  "evidence": ["TypeError 来自 token 为空"],
  "errors": ["login.spec.ts failed"],
  "budget": {"steps_left": 4}
}
```

状态对象适合程序读取和更新，比完整对话更稳定。Runtime 可以基于状态判断是否继续、是否请求用户补充、是否触发人工确认。

### 2.2 上下文窗口

上下文窗口适合放当前轮必需信息。它不适合保存全部历史。长任务中，Runtime 应把原始轨迹保存在外部日志，把摘要和关键证据放入模型输入。

## 3. 长期记忆

### 3.1 可保存内容

长期记忆应保存对未来任务有价值的信息，例如用户偏好、项目约束、稳定业务规则、已验证结论、失败修复经验。临时寒暄、低置信度推测和敏感信息不应随意写入。

### 3.2 读取时机

长期记忆通常在任务开始、计划生成、工具失败和最终回答前读取。读取过早会带来噪音，读取过晚又可能错过关键约束。实践中可以先按任务类型召回少量候选，再由模型或规则筛选。
