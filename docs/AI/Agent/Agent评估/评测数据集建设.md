# 评测数据集建设

## 1. 数据集原则

### 1.1 高质量用例

Agent 评测数据集要来自真实业务任务、真实政策、真实系统状态和真实失败模式。每条用例都应有明确成功标准、可复现环境、评分方式、风险等级和来源版本。

只堆题目数量没有意义。若两个领域专家都无法判断是否成功，这条用例不适合进入发布门禁。若环境不可复现，失败也难以归因。

### 1.2 分层设计

| 数据集 | 目的 | 运行频率 |
| --- | --- | --- |
| Smoke Suite | 验证服务、工具、trace 可用 | 每次提交或部署 |
| Regression Suite | 防止核心能力倒退 | PR、模型升级、prompt 更新 |
| Capability Suite | 衡量新能力和难题爬坡 | nightly、版本评审 |
| Safety Suite | 发现越权、注入、隐私风险 | 发布前、红队 |
| Replay Suite | 复现线上失败 | 事故后、每周回归 |
| Calibration Set | 校准评分器 | 评分器变更、定期审查 |

## 2. 用例结构

### 2.1 必要字段

```json
{
  "case_id": "refund-001",
  "intent": "申请退款",
  "initial_state": {"order_status": "paid"},
  "available_tools": ["get_order", "create_refund"],
  "success_criteria": ["退款状态为 submitted", "金额正确"],
  "forbidden_actions": ["跳过身份校验"],
  "risk_level": "high",
  "source": "production_trace"
}
```

字段要能支持复现、评分和归因。成功标准尽量落到环境状态，例如数据库、API、文件、页面或测试结果。开放文本任务也应有引用、覆盖和限制说明。

### 2.2 来源

数据来源包括业务流程拆解、生产 trace、客服工单、用户反馈、人工接管、红队样本和合成数据。生产 trace 需要脱敏和访问控制。合成样本必须经过抽样审查，不能直接作为高风险门禁。

## 3. 持续维护

### 3.1 失败回流

线上失败应进入 replay suite。处理流程包括：定位低分 trace，脱敏，补齐初始状态和期望结果，选择评分器，加入回归，验证修复。

### 3.2 数据质量风险

| 风险 | 表现 | 控制 |
| --- | --- | --- |
| 任务含糊 | 专家判断不一致 | 增加参考解和成功标准 |
| 评分器过脆 | 替代方案被误判 | 评估 outcome 和关键轨迹 |
| 数据偏斜 | 只测应触发动作 | 加入反样本 |
| 泄漏 | Agent 看到答案 | 隔离环境和答案隐藏 |
| 维护缺失 | 旧用例与新政策冲突 | owner、版本、退役机制 |
