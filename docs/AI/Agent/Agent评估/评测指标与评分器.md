# 评测指标与评分器

## 1. 指标分层

### 1.1 业务优先

Agent 指标应从业务结果开始，再看过程证据、系统质量和安全治理。只看最终文本分数会遗漏工具误用、越权写入、成本失控和状态错误。

| 维度 | 指标 | 评分方式 |
| --- | --- | --- |
| 业务完成 | 任务成功率、最终状态正确率 | 环境断言、接口检查 |
| 策略合规 | 规则遵守率、越权操作率 | 规则评分器、审计日志 |
| 工具使用 | 工具选择准确率、参数有效率 | trace 断言 |
| 检索质量 | groundedness、引用覆盖率 | 引用检查、事实覆盖 |
| 多轮交互 | 澄清能力、状态保持 | 用户模拟器、轨迹分析 |
| 可恢复性 | 失败检测率、重试成功率 | 故障注入、trace replay |
| 安全合规 | PII 泄露率、危险动作阻断率 | 安全分类器、人工审计 |
| 成本性能 | 延迟、token/任务、成本/成功任务 | metrics、账单、trace |

这些指标应按任务类型组合。代码 Agent 重视测试和 diff，客服 Agent 重视策略合规和最终业务状态，研究 Agent 重视来源质量和覆盖度。

### 1.2 硬门禁与加权分

硬门禁用于安全违规、越权写操作、关键业务状态错误和 trace 缺失。加权分用于任务完成、沟通质量、工具效率、引用质量、成本和延迟。高风险任务不应让平均分掩盖严重错误。

## 2. 评分器组合

### 2.1 三类评分器

| 评分器 | 适合内容 | 优点 | 局限 |
| --- | --- | --- | --- |
| 代码/规则 | 数据库状态、schema、权限、测试 | 稳定、便宜、可复现 | 难评开放质量 |
| LLM Judge | 语义覆盖、清晰度、引用质量 | 能处理开放文本 | 需要 Rubric 和校准 |
| 人工评分 | 高风险、争议样本、主观质量 | 判断更稳 | 成本高，速度慢 |

能用程序判断的部分优先使用程序。模型评分器适合补充开放式质量，但要固定模型版本、评分提示和 Rubric，并用人工黄金集校准。

### 2.2 Rubric 设计

Rubric 应明确评分维度、等级定义和失败样例。例如研究报告可以拆成来源可信、事实覆盖、引用对应、结论边界和表达清晰。客服任务可以拆成身份校验、政策遵守、动作正确和用户说明。

### 2.3 评分器漂移

LLM Judge 可能随模型版本、提示和上下文变化产生漂移。应建立 calibration set，定期比较模型评分和人工评分的一致性。关键门禁不要依赖单一 LLM Judge。

## 3. 不同 Agent 的重点

### 3.1 类型对比

| Agent 类型 | 重点指标 |
| --- | --- |
| 客服/销售 | 多轮任务成功、策略合规、人工接管 |
| 代码 | fail-to-pass、pass-to-pass、代码质量、工具轨迹 |
| 研究 | groundedness、覆盖率、来源质量、幻觉率 |
| 数据/BI | SQL 正确性、指标口径、权限和安全 |
| Web/GUI | 页面状态、后端状态、操作路径 |
| 多 Agent | 角色责任、通信质量、冲突处理 |

指标选择应围绕任务风险。低风险内容生成可以多看语义质量，高风险业务操作要优先看最终状态和合规。
